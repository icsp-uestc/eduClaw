# 开发与应用报告

## 一、开发背景

在当前高校教学管理实践中，学生学业指导工作面临着若干亟待解决的现实问题。首先，课程信息、学生成绩、培养方案等关键数据分散在教务系统、电子表格和纸质手册等不同载体中，学生难以通过单一入口获取个性化的学业指导，信息的碎片化直接导致了决策效率的低下。其次，学业预警机制普遍存在滞后性，学生挂科、学分不足等风险往往要到期末成绩公布或毕业资格审核时才会暴露，此时已错过最佳的干预窗口期，补救成本远高于预防成本。第三，高校辅导员和学业导师通常需要面对数十乃至上百名学生，受限于人力配比，难以为每一位学生提供精细化的学业路径分析和持续跟踪。第四，传统的教务查询方式需要登录专门系统、熟悉操作流程，对学生的主动性有较高要求，相当一部分学生因操作门槛而放弃主动获取学业信息。

随着大语言模型技术的快速发展，上述问题的解决路径变得清晰起来。现有的大语言模型已具备工具调用能力，能够根据用户意图动态查询外部数据源并生成结构化的自然语言回复。以 AgentScope 为代表的开源智能体框架支持推理—行动循环模式，可实现"分析用户意图—调用对应工具—解读返回数据—生成回复"的完整闭环。前端技术则能够解析模型输出的结构化标记，将其渲染为可交互的按钮和输入框，从而大幅降低学生的使用门槛。此外，后台定时任务机制可将传统的"被动查询"模式转变为"主动服务"模式，让学业预警在风险发生的当下即刻送达学生手中。基于以上分析，利用 AI 大模型构建一个面向教育场景的伴学智能体，在技术可行性、成本可控性和实际效用方面均已具备充分条件。

## 二、设计与开发

### （一）平台/技术选择

在平台架构方面，本系统采用了前后端分离的设计理念。大模型层面，系统默认接入 DeepSeek v4-pro，该模型提供了与 OpenAI 完全兼容的 API 接口，支持函数调用和推理增强，同时预留了本地 Ollama 模型接入通道，可在离线环境中使用 qwen2.5 等开源模型。智能体框架选用了阿里通义实验室开源的 AgentScope，其 ReActAgent 组件能够编排多轮的工具调用与推理过程，InMemoryMemory 组件提供会话记忆能力，Toolkit 组件支持工具函数的动态注册与发现。后端服务基于 Python Flask 框架搭建，提供 RESTful API 接口供前端调用，所有数据存储采用 JSON 文件格式，包括课程数据库、学生档案、学习路径和预警规则四个独立数据文件，便于非技术人员直接编辑维护。前端界面采用原生 HTML、CSS 和 JavaScript 实现，无任何构建依赖和第三方框架，Chart.js 用于雷达图等数据可视化渲染。整体架构轻量化、可移植性强，单机即可部署全部服务。

### （二）开发过程

本项目的开发采用了"人定架构、AI 编码、人做验收"的协作模式。首先由开发者完成整体技术路线规划，明确四个核心模块（Web 前端、API 网关、智能体框架、Skills 技能库）的边界和职责，选定 DeepSeek v4-pro 作为大模型、AgentScope 1.0 作为智能体框架、Flask 作为后端服务、原生 HTML/CSS/JS 作为前端技术栈，并收集了各框架的开发文档作为 Agent 的工作参考。随后，在每个模块的开发中，开发者先以自然语言向 Agent 描述该模块的功能需求和技术约束，Agent 生成完整的代码实现，开发者运行测试并根据结果反馈修正意见，如此迭代直到模块通过验收。

**Web 前端模块。** 开发者向 Agent 提出需求："构建一个 ChatGPT 风格的聊天界面，包含左侧会话历史栏、中央对话窗口、底部输入框和技能选择器。"Agent 生成完整的 HTML 结构、CSS 样式和 JavaScript 交互逻辑后，开发者启动服务测试。首轮发现技能选择器悬浮菜单的关闭逻辑存在缺陷——点击外部区域不会自动收起，开发者将现象反馈给 Agent，Agent 修正了事件冒泡处理。后续又发现新建对话按钮点击无响应，经排查是 /api/chat/clear 接口的异常未捕获导致后续代码阻塞，Agent 添加 try-catch 保护后修复。经过四轮迭代，前端完成全部交互功能。

**API 网关模块。** 开发者向 Agent 描述了接口体系："需要 /api/chat 处理对话、/api/skills 列出技能、/api/tasks 管理定时任务、/api/permissions 控制权限、/api/auth 处理用户登录。"Agent 生成了 Flask 路由骨架和各端点的业务逻辑。部署测试时发现 /api/chat 的响应模式不统一——使用 LLM 和直接调用工具返回的数据结构不一致，导致前端解析出错。开发者指出问题后 Agent 统一了响应格式并添加 mode 字段标识回复来源。此后又经历了权限审批端点绕过 LLM、任务管理器重复创建默认任务等问题的迭代修复，网关模块逐步收敛稳定。

**智能体框架模块。** 此模块的核心工作是将 AgentScope 的 ReActAgent 与项目 Skills 体系对接。开发者指定约束："系统提示词不能硬编码，必须根据已注册的技能列表动态生成；Agent 需要携带当前登录用户身份信息；与 DeepSeek 的 thinking 模式存在兼容性问题需处理。"Agent 完成了 edu_assistant.py 的编写。第一次集成测试时 Agent 在连续对话中丢失上下文，原因是每个请求都清空了 InMemoryMemory，Agent 将清理逻辑调整为仅在新建对话时执行后修复。第二次测试发现 DeepSeek 的 tool_use 调用完成后无法继续对话，Agent 定位到 reasoning_content 回传缺失问题，在 generate_kwargs 中禁用 thinking 模式后解决。

**Skills 技能库模块。** 开发者向 Agent 提出架构设计："每个技能必须是自包含的目录，包含 SKILL.md（YAML 元数据 + Markdown 指令文档）和实现脚本，系统启动时通过 SkillRegistry 自动发现和注册。"Agent 生成了 SkillRegistry 的扫描加载逻辑和五个技能（课程检索、课程推荐、学习路径、能力画像、学业预警）的完整实现。首轮测试发现能力画像和学业预警技能内部硬编码了学生数据，开发者要求"数据必须从 data/ 目录的 JSON 文件动态加载"，Agent 重构了 get_current_student 调用链并同步更新了任务系统的条件检查器。后续又增加了交互式响应技能作为内置能力，Agent 将其从用户可选技能列表中隐藏并融入了系统提示词。整个 Skills 库的开发过程中，每项技能的增长仅需两个文件，新增的 interactive-response 技能从提出需求到上线通过不超过 30 分钟。

### （三）功能架构

本系统的功能架构由五个核心模块和一个基础设施层构成，采用清晰的层次化设计。

最上层是 Web 交互界面，采用 ChatGPT 风格的聊天布局。左侧为可收起的侧边栏，包含新建对话按钮和历史对话列表。中央主区域为对话窗口，Agent 的回复以气泡形式呈现，其中交互式选项渲染为可点击按钮，能力画像数据渲染为雷达图。底部输入栏左侧集成技能选择器，以图标按钮收纳所有可用技能的弹出菜单，选中的技能以标签形式显示在输入栏上方。页面顶部右侧依次排列用户切换徽章、任务调度铃铛按钮、模型连接状态指示器和权限确认弹窗。

中间层为 Flask API 网关，提供完整的 RESTful 接口体系。chat 端点处理对话请求，根据当前登录用户和所选技能路由到 Agent 或直接调用工具函数；status 端点返回模型连接状态和支持交互测试；tasks 端点管理心跳任务的创建、删除和日志查询；permissions 端点控制各操作的四级权限规则；auth 端点处理用户登录、注销和身份查询；skills 端点返回当前可用的技能列表。此外，conversations 端点在前端管理对话历史，chat/clear 端点在新建对话时清空 Agent 的记忆。

核心层为 AgentScope ReActAgent 智能体。系统启动时完成 Agent 的初始化和模型连接，此后 Agent 实例作为全局单例服务所有请求。每次对话请求到达时，Agent 根据系统提示词中的功能描述和用户意图，决定是否需要调用某个工具函数。调用链路为：用户消息到达 → Agent 推理需要调用的工具 → Toolkit 执行对应的异步函数 → 函数从 data 目录读取 JSON 数据 → 返回 ToolResponse → Agent 基于工具输出生成最终回复。对于需要权限确认的操作（如学业预警），权限管理器在工具调用前拦截请求，返回待确认状态，由前端弹出确认对话框。

数据层包括四个独立的 JSON 文件，分别存储课程库、学习路径、学生档案和预警规则。数据通过 educlaw.core.data_loader 模块统一加载，加载后的数据缓存在内存中，后续请求直接使用缓存。所有技能实现中不再包含任何硬编码的学生数据或课程数据，实现了代码与数据的完全解耦。

在扩展性方面，SkillRegistry 使得新增技能只需在 skills 目录下创建符合规范的子目录和文件即可，无需修改任何注册表或配置文件。前端技能选择器通过 /api/skills 动态获取可用技能列表，弹出菜单自动更新。Agent 的系统提示词同样动态生成，新增技能即刻生效。

## 三、应用过程与效果

本系统已在学院部分计算机科学与技术专业的学生中进行了试点应用。学生通过浏览器访问系统，以自然语言对话的方式完成课程检索、能力画像查看和学业风险自查等操作。试点期间，学生普遍反馈系统操作简便、回复直观，尤其是可点击的按钮式交互和雷达图能力展示获得了较高的认可。多位学生表示，相比传统教务系统的多步查询流程，EduClaw 的自然语言交互方式大幅降低了使用门槛，使学业信息的获取变得轻松自然。

基于试点阶段的积极反馈，项目团队计划在后续推进以下工作：一是将试点范围扩展至学院内其他专业以及校内其他学院，进一步验证系统在不同专业背景下的适用性；二是与兄弟院校合作开展跨校部署测试，收集更多样化的使用反馈；三是将开发经验与应用数据整理为教学研究论文进行发表，同时在全国性的教学信息化会议中进行成果展示和经验推广，让更多一线教师了解 AI 智能体辅助教学管理的可行路径。

## 四、创新与反思

### （一）创新点

本项目的创新并非体现在使用了某种前沿模型，而在于将 AI 智能体的标准化架构真正适配到教育场景，并在多个维度上给出了可复现的工程方案。

**第一，文件系统驱动的技能扩展机制。** 不同于 Dify、Coze 等低代码平台的图形化插件机制，EduClaw 的技能体系严格遵循 Anthropic 发布的 Agent Skills 开源规范。每个技能是一个自包含的独立目录，其中 SKILL.md 文件以 YAML 前置元数据记录技能的名称和用途，正文详细描述触发条件、执行流程和输出示例，完全可以作为一份产品规格书来阅读。Python 实现脚本则负责具体的数据查询逻辑。SkillRegistry 组件在系统启动时扫描 skills 目录下的所有子文件夹，自动完成模块加载和工具注册。Agent 的系统提示词由已发现的技能列表动态拼接生成，前端技能选择器通过 API 实时获取可用功能列表。这种设计意味着新增一个功能仅需创建目录并编写两个文件，无需修改任何注册表、配置文件和前端代码，扩展过程具有天然的版本可追踪性。对于教学团队而言，教师负责编写 SKILL.md 描述教学需求，技术人员负责实现对应的查询逻辑，两者通过文件系统完成协作，而非依赖特定平台的权限体系或工作流编辑器。

**第二，从被动响应到主动服务的任务调度创新。** 传统智能体应用通常仅在被用户提问时做出响应，而 EduClaw 内置的双模任务调度引擎实现了主动监测能力。ScheduledTask 按照预设时间间隔自动执行指定提示词，适用于每日学习建议、定期学业总结等周期性场景。TriggerTask 的核心是一组可插拔的条件检查器函数，每个函数轮询学生数据中的特定字段——例如 credit_shortage 条件检查器遍历全部学生档案，当学分缺口超过 10 分时返回触发上下文；low_gpa 检查器在 GPA 低于 2.5 时激活。TaskManager 以后台守护线程的形式每十秒轮询所有任务，对满足条件的任务调用 Agent 执行预设提示词并持久化结果。条件检查器通过 CONDITION_REGISTRY 注册，未来接入缺课次数、作业提交率等更复杂的监测规则时，无需修改调度引擎的核心逻辑。这种"条件触发→Agent 执行→推送结果"的闭环，将学业预警从事后通知转变为事中干预，在高校学业管理的实际场景中具有直接的实用价值。

**第三，交互协议层的前端渲染创新。** 聊天机器人领域长期存在一个体验困境：当需要用户从多个选项中选择时，模型只能以文本罗列选项并要求用户手动打字回复，这不仅效率低下，而且用户的输入格式往往不规范导致模型误解。本项目的解决方案是定义了一套基于 XML 风格标签的交互协议。当 Agent 需要学生做出选择时，在回复中嵌入 `<action type="select">` 标签包裹的 Markdown 表格；当需要学生输入信息时，嵌入 `<action type="input">` 标签。前端 JavaScript 在渲染回复时扫描并提取这些标签，将其替换为可点击的按钮组或输入框组件。学生从传统的"阅读文字→理解选项→打字回复"三步操作，简化为"看到按钮→点击"一步完成。Agent 通过 InMemoryMemory 维持完整的多轮对话上下文，Agent 能够理解学生点击的按钮值对应的是哪一轮对话中的哪个问题。这种将大模型的文本输出能力与前端组件渲染能力通过标准化标签语法桥接的设计，本质上构建了一种"对话即界面"的新范式，大幅降低了自然语言交互中的歧义和操作门槛。

**第四，数据全解耦的架构设计。** 当前诸多教育类 AI 工具的数据与代码紧密耦合，使得非技术人员无法独立维护业务数据。本项目将所有数据从代码中完全分离，课程库、学生档案、学习路径和预警规则分别存储在四个独立的 JSON 文件中，通过统一的数据加载器模块读取并缓存。教学管理人员只需用文本编辑器修改这些 JSON 文件即可完成课程增删、学生信息更新、预警阈值调整等操作。三个预置的学生角色 Demo、Alice 和 Bob 分别代表了"中等生""预警生"和"优秀生"的典型画像，覆盖了系统核心功能验证所需的全部场景。数据层还通过 Flask session 机制实现了严格的用户数据隔离，get_current_student 函数从当前会话中获取登录用户的身份标识并限定数据查询范围，从根本上杜绝了跨用户数据泄露的可能性。

**第五，细粒度的伦理合规设计。** 四级权限模型（始终允许、会话内确认一次、每次请求确认、全局拒绝）将安全控制的粒度精确到每个技能操作。学业预警等涉及学生隐私的高风险操作默认为"每次确认"，课程检索等日常查询为"始终允许"，用户可通过 Web 界面实时调整任意技能的权限级别。Agent 的系统提示词中明确注入了禁止跨用户数据泄露的安全原则。这些设计与《生成式人工智能服务管理暂行办法》的要求以及"创AI"案例评价标准中"注重人工智能伦理和责任落实"的导向完全一致。

### （二）与同类工具的差异化

当前教育智能体的主流开发路径是使用 Dify、Coze 等低代码平台进行可视化搭建。EduClaw 与这类方案存在本质差异。在部署方式上，低代码平台依赖平台服务且数据存储于第三方服务器，而 EduClaw 基于纯 Python 生态，单机即可运行全部服务，学生数据完全本地化，不存在跨境传输或第三方访问的风险。在扩展方式上，低代码平台依赖图形化拖拽工作流，配置难以版本化管理，而 EduClaw 以文件系统驱动的目录结构天然兼容 Git 等版本控制工具，每一次技能变更都有据可查。在交互体验上，纯文本对话模式下学生需要主动记忆和输入指令，而 `<action>` 标签协议实现了点击即交互。在数据隐私上，低代码平台通常由平台方持有全部交互日志，而 EduClaw 的本地 JSON 方案保证了学生成绩等敏感数据不离开校园网络。

### （三）问题与改进

在开发过程中遇到了若干值得记录的技术挑战。DeepSeek 模型的推理增强功能与 AgentScope 的消息格式化器存在兼容性问题——后续请求需要回传 reasoning_content 字段但格式化器会丢弃该类型消息块，导致第二轮工具调用时 API 返回 400 错误，解决方案是在请求参数中显式禁用 thinking 模式。Agent 记忆管理也经历了多轮策略调整，记忆过长导致上下文膨胀和推理延迟增加，记忆过短导致丢失对话连贯性，最终采用"单会话保留、新建对话清空"的折中方案。Python 包管理中的命名冲突也值得注意：Flask 实例变量名 app 与模块文件名 app.py 重名，导致 import 语句意外解析为 Flask 对象而非模块，通过规范 __init__.py 的导出范围得以解决。

展望未来，数据层的升级是最高优先级的演进方向，从 JSON 文件迁移至 SQLite 或 MySQL 将支撑更大规模的真实部署。多模态能力的接入将允许学生拍照上传成绩单等纸质文档并通过 OCR 自动录入。流式输出的引入将显著改善交互体验，使模型回复内容实时呈现而非一次性返回。通过适配 Telegram、微信和钉钉等平台的 Bot 接口，可让学生在日常使用的社交工具中直接获得学业指导。学习数据的长期积累还可用于生成班级乃至院系层面的学情分析报告，为教学管理决策提供数据支撑。
